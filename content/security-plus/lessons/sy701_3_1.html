Exam objective 3.1

Compare and contrast security implications of different architecture models

Different architecture models have distinct security implications, affecting how security controls are implemented and managed.

Architecture and Infrastructure Concepts

  • Cloud: Utilizing third-party cloud providers (e.g., AWS, Azure, Google Cloud) for computing resources. Security is a shared responsibility model.
    • Responsibility matrix: Defines the security responsibilities between the cloud provider and the customer.
    • Hybrid considerations: Integrating on-premises infrastructure with cloud services, requiring careful security planning for connectivity and data flow.
    • Third-party vendors: Relying on external vendors for services, necessitating vendor risk management.
  • On-premises: Hosting all IT infrastructure within the organization's own facilities. Full control over security, but also full responsibility.
  • Centralized vs. Decentralized:
    • Centralized: Security management and controls are managed from a single point. Can offer consistency but also a single point of failure.
    • Decentralized: Security management and controls are distributed across different teams or locations. Can offer flexibility but may lead to inconsistencies.
  • Containerization: Packaging applications and their dependencies into isolated containers (e.g., Docker, Kubernetes). Offers portability and efficiency, but requires container-specific security measures.
  • Virtualization: Creating virtual versions of computing resources (e.g., servers, operating systems, storage). Offers resource utilization and flexibility, but introduces new attack surfaces (e.g., VM escape).
  • IoT (Internet of Things): Network of physical objects embedded with sensors, software, and other technologies for connecting and exchanging data over the internet. Often have limited security features and are vulnerable to attacks.
  • Industrial Control Systems (ICS)/Supervisory Control and Data Acquisition (SCADA): Computerized systems that control industrial processes. Often critical infrastructure, requiring specialized security measures due to their operational nature and potential for physical impact.
  • Real-time Operating System (RTOS): Operating systems designed for applications with strict timing requirements. Security must not compromise real-time performance.
  • Embedded Systems: Computer systems designed for specific functions within a larger mechanical or electrical system. Often have limited resources and may lack robust security features.
  • High Availability: Designing systems to minimize downtime and ensure continuous operation. Security measures should not hinder availability.

Infrastructure Considerations

  • Infrastructure as Code (IaC): Managing and provisioning infrastructure through code instead of manual processes. Improves consistency and reduces human error, but requires secure coding practices for infrastructure definitions.
  • Serverless: Cloud execution model where the cloud provider dynamically manages the allocation and provisioning of servers. Reduces operational overhead but shifts some security responsibilities to the cloud provider.
  • Microservices: An architectural style that structures an application as a collection of loosely coupled, independently deployable services. Increases complexity in securing inter-service communication.
  • Network Infrastructure:
    • Physical isolation / Air-gapped: Completely separating a network segment from other networks, including the internet, for maximum security.
    • Logical segmentation: Dividing a network into smaller, isolated segments using VLANs, subnets, or firewalls to control traffic flow and limit the impact of breaches.
    • Software-defined networking (SDN): Decoupling the control plane from the data plane, allowing for centralized and programmable network management. Offers flexibility but introduces new security considerations for the control plane.

Considerations (General Security Implications)

  • Availability: Ensuring systems and data are accessible when needed. Security measures should not negatively impact availability.
  • Resilience: The ability of a system to recover from failures and continue operating. Security architecture should incorporate resilience.
  • Cost: The financial investment required for implementing and maintaining security measures.
  • Responsiveness: The ability of security systems to detect and respond to threats quickly.
  • Scalability: The ability of security solutions to handle increasing demands and growth.
  • Ease of Deployment: How easily security solutions can be implemented and integrated into existing infrastructure.
  • Risk Transference: Shifting the financial burden of risk to another party, often through insurance.
  • Ease of Recovery: How quickly and efficiently systems and data can be restored after an incident.
  • Patch Availability: The frequency and reliability of security patches from vendors.
  • Inability to Patch: Situations where systems cannot be patched due to legacy issues, vendor support, or operational constraints.
  • Power: Ensuring reliable power supply for security systems and critical infrastructure.
  • Compute: The processing power and resources required for security operations, such as encryption, logging, and analysis.
